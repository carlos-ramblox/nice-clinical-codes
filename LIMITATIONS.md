# Limitations

## Framework

Wood and Higgins (Bennett Institute for Applied Data Science) published a blog post in September 2023 titled [*What Are Codelists and How Are They Constructed?*](https://www.bennett.ox.ac.uk/blog/2023/09/what-are-codelists-and-how-are-they-constructed/), in which four common failure modes in codelist construction are identified. This document maps the clinicalcodes.uk pipeline against each of those failure modes. It is intended as an honest account of where the pipeline mitigates the risk and, more importantly, where it does not.

The pipeline parses a natural-language query into structured conditions, runs four retrievers in parallel (OMOPHub, OpenCodelists, QOF Business Rules, and a ChromaDB semantic index over ingested vocabulary), enriches the merged candidate set against UMLS Metathesaurus relationships, scores each candidate with Claude Haiku 4.5 at temperature 0, and routes the result through a clinician review gate before any artefact is signed and exported.

## Failure mode 1: Including similar-sounding but unrelated codes

Bennett's example: "ocular hypertension" being mistakenly included in a hypertension codelist.

**How the pipeline addresses this.** Each candidate is scored by Claude Haiku 4.5, which returns an `include`/`exclude`/`uncertain` decision, a confidence score, and a one-sentence rationale; all three are stored in the audit log. The scoring prompt explicitly instructs the model to exclude codes for unrelated comorbidities and to mark ambiguous cases as `uncertain` rather than silently including them. Candidate provenance (which retriever surfaced the code, how many sources support it) is preserved through to review, so a reviewer can see at a glance whether a code is corroborated by multiple vocabularies.

**Residual risk and gaps.** The LLM rationale is a heuristic, not a proof; the pipeline does not formally guarantee that similar-sounding unrelated codes are excluded. There is no curated adversarial test set specifically targeting this failure mode — the [EVALUATION.md](./EVALUATION.md) benchmark compares against published OpenCodelists, which does not isolate this class of error. The reviewer must still verify exclusions manually.

**Closest current proxy: offline RAGAS-style faithfulness check (T11).** The set-based F1 metric in [EVALUATION.md](./EVALUATION.md) does not detect a wrong-but-similar-sounding inclusion when the wrong code happens to share a term substring with the queried condition. As the closest available proxy for this failure mode, an offline LLM-as-judge groundedness check is run across all 1,329 (codelist × scored code) pairs in the v2 benchmark; see [EVALUATION.md → §Faithfulness](./EVALUATION.md#faithfulness). Mean groundedness 83.5 %; 2.2 % of rationales are flagged `unfounded` by the judge. The check is offline-only (not on the live `/api/search` path), is single-judge (Haiku 4.5; carries the self-preference caveat documented in §Faithfulness → *Known biases*), and is not a substitute for a curated adversarial test set or for clinician review. It is reported as the most direct quantitative signal currently available against this failure mode.

## Failure mode 2: Omitting synonyms

Bennett's example: a sore-throat codelist failing to include codes for pharyngitis.

**How the pipeline addresses this.** UMLS Metathesaurus enrichment expands each top-ranked candidate with its synonyms (atom-level string variants of the same CUI), narrower concepts (`RN` relations), and sibling concepts (`SIB` relations). Running four retrievers in parallel further reduces the chance that a synonym held in one vocabulary but not another is missed.

**Residual risk and gaps.** UMLS coverage is itself incomplete, and the enrichment step is capped at the top 30 ranked candidates per query, so tail-end synonyms may not be surfaced. There is no automated check that all canonical synonyms for a concept have been considered; the reviewer must still confirm obvious synonyms are present.

## Failure mode 3: Misunderstanding study intent

Bennett's example: whether gestational diabetes belongs in a diabetes codelist depends on the specific research aims.

**How the pipeline addresses this.** The query parser converts free-text input into structured conditions that are passed verbatim to the scoring step, which now (T29) includes per-condition `include_criteria` and `exclude_criteria` lists. The parser extracts these from natural-language phrasings ("X excluding Y", "X but not Y", "X without Y", "X, exclude Y", "X, not Y") and the search request also accepts structured `inclusions` / `exclusions` arrays for callers that prefer an explicit UI escape hatch. The per-code scoring prompt is instructed to set `decision="exclude"` for any code whose meaning falls under an exclude criterion, naming the matched term verbatim in the rationale so the carve-out is auditable. Criteria are persisted on the codelist and conditionally appended to the SHA-256 signature on approval (see [EVALUATION.md §5.9](./EVALUATION.md#59-study-intent-test-case-t29) and `backend/tests/test_signature_hash_criteria.py`); the conditional-append rule preserves byte-for-byte compatibility with every pre-T29 approved hash, so existing artefacts continue to verify under the new code path. Reviewer overrides at the per-code stage are captured with a written comment in the audit log and contribute to the same signature.

**Residual risk and gaps.** The scoring prompt still encodes a small set of hardcoded study-intent heuristics for known cases (for example, the type 1 vs. type 2 diabetes distinction and the diabetic-retinopathy carve-out); these remain baked into the prompt and not configurable per-query. Criterion application is heuristic — the LLM is instructed to match by clinical meaning, but there is no formal proof that any given exclusion term suppresses every code it semantically covers. The fixture at `data/test_sets/persona_audit/diabetes_no_gestational.json` exercises the gestational case but a broader adversarial test set across study-intent variants has not yet been curated. Mitigation is now first-class but remains, of the four failure modes, the one most dependent on LLM judgement at scoring time.

## Failure mode 4: Codes that refer to both relevant and irrelevant cases

Bennett's example: "sore throat" indicates Group A Streptococcus but also many other conditions — improving sensitivity costs specificity.

**How the pipeline addresses this.** The per-code rationale captures, in one sentence, why the model judged the code relevant; this is visible to the reviewer alongside source provenance. Codes the model flags `uncertain` are routed for explicit reviewer adjudication rather than being assigned a default.

**Residual risk and gaps.** There is no sensitivity-vs-specificity preference toggle: the pipeline returns what the model scores, and the reviewer cannot ask the system for a "high-sensitivity superset" or a "high-specificity consensus" variant of the same query. The trade-off must be applied by the reviewer at the review stage and is not captured as a structured pipeline parameter.

## Limitations beyond the Bennett framework

- Evaluation reference sets are derived from existing OpenCodelists curations, but the mapping between each reference codelist and the corresponding test query was interpreted by a single non-clinician evaluator; no independent peer validation of that selection has been performed. The F1 numbers in [EVALUATION.md](./EVALUATION.md) should be read as pilot-tier signal on n = 15 codelists rather than a peer-reviewed methods evaluation.
- Sample composition is skewed toward `nhsd-primary-care-domain-refsets` (12 of 15 codelists). The QOF retriever is built from the same NHS QOF Business Rules from which those refsets are derived, so the sample is favourable to QOF retrieval and may overstate QOF's marginal contribution at scale. Full caveat in [EVALUATION.md → Methodology → Sample selection](./EVALUATION.md#sample-selection).
- Recall is the system's primary current weakness. Mean recall across the 15 reference codelists is 0.50; the LLM scoring step trades recall for precision (precision 0.62 → 0.86; recall 0.62 → 0.50). The recall ceiling is set by the four retrievers' joint coverage on each query and is decomposed by retriever in [EVALUATION.md §Per-retriever ablation](./EVALUATION.md#per-retriever-ablation). Improving recall is the next research direction (retrieval reranking, query reformulation, additional retrievers).
- Codelists are point-in-time. SNOMED releases, QOF rule changes, and OpenCodelists edits do not trigger automatic re-validation of previously approved codelists.
- OpenCodelists has no documented public REST API. The pipeline ingests OpenCodelists content via stable URL/CSV permalinks committed under `data/raw/opencodelists/`. This works in practice but there is no formal contract from OpenCodelists guaranteeing permalink stability; integration is best-effort and may need adjustment if OpenCodelists changes their URL schema.
- No multi-condition cohort composition (for example, "diabetes AND hypertension AND age > 65" as a structured query).
- HDR UK Phenotype Library integration is deliberately scoped to read-mode only (discovery sidebar + post-hoc cross-reference, T34 + T35); it is not a parallel retriever. See [EVALUATION.md §5.7 → HDR UK retriever-shape mismatch](./EVALUATION.md) for the negative-result write-up that established that integration shape.
- **Code-usage column (T31).** Each result carries `usage_frequency`, `usage_status`, `usage_source` and `usage_setting`, populated from NHS Digital's primary-care SNOMED reporting (Aug 2024 – Jul 2025) and HES inpatient ICD-10 / OPCS-4 reporting (Apr 2024 – Mar 2025). Methodology follows Bennett Institute's [OpenCodeCounts](https://bennettoxford.github.io/opencodecounts/); we ingest the upstream NHS Digital CSVs directly under Open Government Licence v3.0 to avoid licence ambiguity on Bennett's harmonised dataset. SNOMED counts are **rounded to the nearest 10** with **counts of 1-4 withheld** under NHS Digital's privacy policy and surfaced as `usage_status="withheld_below_5"` (UI renders "<5"); ICD-10 / OPCS-4 are unrounded with zero-usage codes excluded by the source. Codes absent from any usage publication return `usage_status="not_in_dataset"` (UI renders "—") rather than a misleading zero. The signal is purely additive — surfaced for reviewer judgement, not used by the LLM scoring step. Specific limits worth stating explicitly:
  - **Geographic scope is England-only.** SNOMED counts are NHS England primary care; ICD-10 / OPCS-4 are HES inpatient England. Codes used exclusively in Wales, Scotland, or Northern Ireland appear as `not_in_dataset` regardless of real-world prevalence.
  - **Care-setting scope is GP + HES inpatient only.** Outpatient, A&E, mental-health, and community settings are not in the source datasets. A code used heavily in outpatient clinics but not by GPs and not in inpatient FCEs will appear as `not_in_dataset`.
  - **SNOMED rounded values are ranges, not points.** A returned `12,400` means the underlying count lies in `[12,395, 12,404]` after NHS Digital's nearest-10 rounding. Do not treat the displayed value as exact. The column-header tooltip cites the rule.
  - **Withheld counts contribute no quantitative signal.** `<5` only tells the reviewer the code was used at least once; no ordering between two `<5` codes is implied.
  - **Annual reporting cadence creates a publication lag.** Codes that became active after the source publication's coverage end (post-Jul 2025 SNOMED, post-Mar 2025 ICD-10 / OPCS-4) appear as `not_in_dataset` until the next NHS Digital release is ingested.
  - **HES counts are primary-position only.** NHS Digital's published HES dataset reports primary-diagnosis (`DIAG_4_01`) and primary-procedure (`OPERTN_4_01`) finished consultant episodes; secondary-position uses are not in the source. A code that frequently appears as a *secondary* diagnosis (e.g. type 2 diabetes mellitus on an admission for a different reason) will report a count substantially smaller than its true any-position prevalence. Treat the HES number as a lower bound for diagnostic prevalence and an accurate measure of "this code drove the admission".
- **OHDSI concept-set export.** `GET /api/export/{search_id}?output_format=ohdsi` and `GET /api/codelists/{id}/export?format=ohdsi` emit the OHDSI Concept Set Specification shape so a codelist round-trips into ATLAS / DARWIN-EU `CodelistGenerator` by paste. Specific limits worth stating explicitly:
  - **`concept_id` coverage depends on OMOPHub.** OMOPHub returns the concept_id natively for the codes its retriever surfaces; the merger then runs a `concept_id_enricher` node that issues a parallel `concepts.get_by_code` lookup for any post-merge candidate without a concept_id (typically those only QOF, OpenCodelists, or ChromaDB found). Lookups are cached process-wide. On a representative `type 2 diabetes` query this lifts coverage from ~19% to ~71%; the residual 29% are codes OMOPHub itself cannot resolve to an OMOP standard concept (UK SNOMED-CT extensions, recent additions, or codes outside the standard vocabulary). Those rows go to the export's `unmapped` array rather than be guessed.
  - **No bidirectional round-trip.** The export is one-way: paste into ATLAS works, but importing a concept set from ATLAS back into clinicalcodes.uk is not implemented. Future work.
  - **No per-code `includeDescendants` flag.** Every item in the export carries `includeDescendants: false` and analysts opt in to descendants via the ATLAS UI after paste. This matches the OHDSI default-false convention but means a clinicalcodes.uk reviewer cannot mark a code as "include with descendants" before export. Future work.
- **OpenCodelists upload-CSV export.** `GET /api/codelists/{id}/export.opencodelists.csv` returns a ZIP with one `code,term` CSV per coding system plus a `provenance.json` carrying the SHA-256 signature, reviewer pair, and agreement κ. The artefact is gated on `status='approved'` AND signature v2 (two-reviewer Delphi). Specific limits worth stating explicitly:
  - **Manual upload only.** OpenCodelists has a token-auth API (`codelists/api.py::create_codelist` in `opensafely-core/opencodelists`) but it is undocumented and tokens are not publicly issued. The export is a download + manual web-form upload via OpenCodelists's "Create a codelist" form; we do not POST on the user's behalf.
  - **Format verified against source, not against a live upload.** The CSV column set (case-sensitive `code` header, optional `term`, no `vocabulary` column) was validated against `codelists/forms.py::CSVValidationMixin` in the OpenCodelists source. We have not performed an end-to-end upload against `opencodelists.org` to confirm bytes-on-the-wire success.
  - **One CSV per coding system.** OpenCodelists scopes each codelist to a single coding system (the `coding_system_id` form field), so a mixed-vocabulary clinicalcodes.uk codelist exports as N CSVs and uploads as N OpenCodelists codelists. The provenance manifest tells the uploader which slug to pick on the form for each file.
  - **Off-corpus vocabularies are dropped, not mislabelled.** Rows whose vocabulary has no OpenCodelists slug — UMLS CUIs, QOF Business Rules, and US ICD-10-CM (which is structurally distinct from WHO ICD-10) — surface in `provenance.json::dropped_codes` with their original vocabulary label, rather than being silently aliased onto a similar-looking slug.
  - **`provenance.json` is not part of the OpenCodelists upload.** OpenCodelists's form takes only the CSV; the provenance trail is for downstream verifiers (and for clinicalcodes.uk's own audit) and survives by sitting alongside the CSV in the ZIP. A user mirroring the codelist into OpenCodelists will need to attach provenance separately if their downstream workflow requires it.
- **Public gallery (T32) is default-public, opt-out per row.** Once a codelist is approved its redacted projection appears at `/gallery` automatically. Reviewer / creator identities are reduced to initials, override rationales are dropped, and UMLS-suggestion rows are filtered out — the public copy is intentionally a different artefact from the audit copy. The author can retract a row at any time via the "Hide from public gallery" toggle on the codelist detail page (mapped to `PUT /api/codelists/{id}/privacy`); retraction itself is recorded in the audit log so the visibility history is reconstructible. Default-public is intentional for *new* codelists: the audit said the value of the HITL signature is invisible to a non-logged-in visitor, and an opt-in default would reproduce that gap. **Pre-T32 codelists are back-filled to `private=1` on migration**, however, so the visibility change is not a one-way door for reviewers who approved their codelists under the older mental model — they keep their original opt-in posture and can flip individual rows public from the codelist detail page if and when they choose. Sites that want a private-by-default deployment for new rows too must change the column DEFAULT in `_init_schema` and add a separate publish step in the review flow. Specific limits worth stating explicitly:
  - **The original query string and per-code AI rationales travel with the public copy.** Both are part of the artefact's meaning (the rationale is the reviewer's main signal for *why* a code is in the list) so they cannot be redacted without making the gallery copy useless. A reviewer who bakes identifying detail into their search query — for example a Trust name or a cohort identifier — should expect that text to appear at `/gallery/{id}` once the codelist is approved. The "Hide from public gallery" toggle is the immediate retraction path.
  - **Public CSV vs OHDSI export defaults differ from the auth-side route.** `GET /api/public/codelists/{id}/export` defaults to `format=csv` because public visitors are more likely to want a quick CSV than an ATLAS-shaped concept set; the auth-side `GET /api/codelists/{id}/export` defaults to `format=ohdsi`. Both formats are reachable on both routes via the explicit `?format=` parameter.
  - **No rate limiting on the public endpoints.** A scraper could enumerate `/api/public/codelists` and pull every CSV in a loop without authentication. The `/count` response carries a short `Cache-Control` window but `/list`, `/get`, and `/export` do not. Acceptable at the current demo scale; deploy behind a reverse proxy with per-IP request limits (or move to a hosted gateway with built-in rate limiting) before opening the surface to broader traffic.
- **Two-reviewer Delphi review (T30).** Codelists can route through a Watson 2017 Stage 3 two-reviewer flow with Cohen's κ surfaced on the review surface; see [`CLINICAL_SAFETY.md`](./CLINICAL_SAFETY.md) for the hazard mapping and [`EVALUATION.md`](./EVALUATION.md) §Inter-rater agreement for the methodology. Specific limits worth stating explicitly:
  - **Cohen's κ is unweighted.** Weighted forms (linear, quadratic) treat include↔uncertain disagreement as less severe than include↔exclude, which has clinical merit but forces a defensible-but-arguable weight choice. Watson 2017 uses unweighted, so we ship that first; the docstring on `cohen_kappa` flags weighted variants as the natural extension.
  - **n=2 only.** The API caps `reviewer_ids` at exactly two; n≥3 (Fleiss' κ) is the natural extension and is rejected at the route layer today (`AssignReviewersRequest.reviewer_ids` `min_length=2, max_length=2`).
  - **No confidence interval on κ.** SE(κ) and bootstrap CI are deferred. The methods-paper deliverable (a real inter-rater study with 2-3 clinicians) is what unlocks the calendar for adding them; not a demo requirement.
  - **Consensus is both-ACK; no admin override.** A single "lead reviewer" with casting vote was considered and rejected on clinical-safety grounds — it would let one reviewer commit the other's clinical agreement to the audit log without their explicit acknowledgement. The `/consensus` route therefore validates byte-equal resolutions between proposer and acknowledger.
  - **Independent voting assumes reviewers don't communicate out-of-band.** The system hides reviewer A's votes from reviewer B until A finalises (Watson 2017 anchoring-bias guard), but cannot enforce that the reviewers don't compare notes via Slack, email, or in person. The independent-voting property is a clinical-process commitment captured by the design, not a technical guarantee.
  - **κ < 0.41 banner is informational, not gating.** The UI surfaces an amber strip and a "consider whether further discussion is needed before consensus" caption when κ falls below the Landis & Koch "moderate" boundary, but the workflow does not block approval at any threshold. Gating approval at a κ value would be a clinical-safety policy decision; the implementation leaves reviewer judgement intact.
  - **`signature_version` is immutable post-creation.** A v1 (single-reviewer) codelist cannot be promoted to v2 (two-reviewer) in place — the version is fixed when reviewers are first assigned. Promotion would invalidate every prior verification of the v1 signature, so the workflow forks instead. Adding reviewers to a legacy codelist creates a new v2 codelist with a new id rather than mutating the existing row.
- **UK drug retrieval (T37).** dm+d and BNF/pseudoBNF are surfaced as two extra parallel retrievers, gated on the parser flagging the query domain as `Drug`. Drug rows merge into the same results table and signature/export chain as disease rows, plus a `dmd_level` badge per dm+d row (Ingredient / VTM / VMP / AMP, inferred from the dm+d preferred term shape — the OpenCodelists CSV export does not carry the level natively). Specific limits worth stating explicitly:
  - **`dmd_level` is a term-shape heuristic, not an authoritative dm+d hierarchy lookup.** Branded products with no `(...Ltd)`-style holder token in the term will be classified as VMP rather than AMP; non-standard term formatting from a third-party CSV may misclassify rows. The badge is a reviewer hint, not a primary-key into dm+d.
  - **No BNF↔dm+d cross-walk surfaced today.** When a query returns both vocabularies the merger deduplicates by `(code, vocabulary)` so a BNF code and its dm+d cross-walk appear as two rows. NHSBSA publishes a cross-walk file; ingesting it for a per-row cross-reference field is a follow-on ticket.
  - **Drug rationales reuse the disease scoring prompt.** FR-007 asks for drug-domain semantics (route, strength, formulation) in the LLM rationale; the prompt is unchanged in T37 and faithfulness on drug rows is unmeasured against the 83.5% disease baseline. If a re-measurement drops below 75% a drug-specific prompt branch becomes a follow-on ticket.
  - **`dmd_level` is not persisted in `codelist_decisions`.** The schema is unchanged; the field is derived at read time from `term`. A future authoritative ingest of the dm+d hierarchy file would let us persist it, but the heuristic is sufficient for the badge today.
  - **Source CSV coverage is sample-tier, not full corpus.** The repo ships representative dm+d / BNF CSVs under `data/raw/opencodelists/{dmd,bnf}/`; a production deploy should pull the full OpenCodelists drug-codelist corpus (or NHSBSA dm+d weekly TRUD release) before relying on recall.
- **Query-intent-aware hierarchy expansion (T37j).** The OMOP "Is a" descendant expander introduced in T37i ran on every query and produced a bimodal F1 signature: descendant-closed gold lists gained +0.16 to +0.38, descendant-pruned gold lists (NICE-style diagnosis-only registers) lost −0.32 to −0.47 (see [EVALUATION.md → T37i](./EVALUATION.md)). T37j gates the expander on a per-query `include_descendants` boolean. The LLM parser sets a default from cues in the query string (`"all forms of X"` → True; `"X diagnosis only"` → False; bare-name queries default to False), and a `request_include_descendants` field on the search request body lets the reviewer override that default from a UI checkbox. The False default and the override together return the disease benchmark to within K=5 σ ≈ 0.012 of the pre-T37i baselines for diagnosis-only codelists while keeping the lift available on descendant-closed ones; see `data/test_sets/benchmark_2026_04/T37j_path_a_summary.md` for the K=5 numbers. Specific limits worth stating explicitly:
  - **LLM cue extraction is heuristic.** A query that should signal True but uses unrecognised phrasing ("any kind of epilepsy", "every flavour of dementia") will extract False, leaving the reviewer to flip the checkbox. The system prompt names six cue patterns explicitly; the long tail is not enumerated and not formally tested.
  - **The decision is made at search time and frozen on save.** Once a codelist is saved with `include_descendants=true`, re-expanding (or un-expanding) the same artefact requires re-running the search. A future re-expand action on the codelist detail page is plausible follow-on work, but T37j keeps the decision at the search-time boundary so the signature stays a function of a single committed state.
  - **The flag enters the v2 signature payload conditionally on True.** When `include_descendants` is False the signature is byte-identical to the pre-T37j payload (T29 conditional-append pattern), so every pre-T37j approved hash continues to verify. When True the payload gains a `\n--descendants--\ntrue` block, locking the routing decision into the codelist's audit chain.
  - **Two K=5 outliers remain inside per-codelist std, not σ.** `dementia` drifted −0.012 against the post-T37i lift and `lung_cancer` drifted −0.054; both sit inside the per-codelist K=5 standard deviation documented for those lists (`hypertension` carries a similar 0.025 std in the run-variance benchmark) but only the first sits inside the K=5 σ ≈ 0.012 aggregate noise floor. The drift is bounded; a third K=5 sweep would tighten the CI without changing the verdict.
- Authentication is a demo mechanism; NHS OAuth/SAML and NHS Identity integration are not yet implemented.
- No clinical safety case has been written. The system is a research and development prototype.
- FAIR-compliance of generated codelists has not been formally evaluated against the criteria set out in Williams et al. (2019).
- Cost and latency at scale (large query batches, multi-tenant load) have not been characterised. Per-query latency is sensitive to model availability, candidate volume, and source-API response time; the README's "tens of seconds" framing is representative of the test conditions used during development and is not a guaranteed bound. Per-query cost is recorded at request time but has not been benchmarked against a fixed test set, so headline cost-reduction figures are not currently claimed.

## Candidate-cap recall ceiling

The merger ([`result_merger.py`](./backend/app/graph/nodes/result_merger.py)) and UMLS enrichment node ([`umls_enrichment_node.py`](./backend/app/graph/nodes/umls_enrichment_node.py)) both truncate the candidate pool to `MAX_CANDIDATES=100` (env-configurable, [`config.py`](./backend/app/config.py)) before LLM scoring. Sorted by `(-source_count, -similarity_score, vocabulary, code)`, the cap fires after dedup and the requested-vocabulary filter but before reasoning, so the LLM never sees codes that rank below position 100 in the merged pool.

Nine of the 15 benchmark codelists carry gold lists larger than the cap and are therefore subject to a structural recall ceiling at `MAX_CANDIDATES=100` (max recall = 100 / gold size): `epilepsy` (0.21), `lung_cancer` (0.28), `dementia` (0.31), `stroke` (0.38), `hiv` (0.41), `psychosis_schiz_bipolar` (0.51), `asthma_pincer` (0.81), `hypertension` (0.85), `depression` (0.94). The 6 smaller codelists are not bounded by this ceiling but the cap can still drop individual gold codes when the pre-cap pool exceeds 100 (`heart_failure` loses ~5 of 42 gold codes to the cap at the validation run; see the sweep diagnostic counts).

The cap-sensitivity sweep ([`cap_sensitivity_summary.md`](./data/test_sets/benchmark_2026_04/cap_sensitivity_summary.md)) measured F1 across `MAX_CANDIDATES ∈ {100, 500, 1000}` paired K=5 on all 15 codelists, with mode-matched aggregation (bare for 8 non-descendant-closed, override for the 7 descendant-closed). Headline:

- **cap=100 mode-matched** (K=5): mean F1 = **0.569**, BCa 95 % CI [0.414, 0.690]. Reproduces the April K=1 0.57 baseline.
- **cap=1000 mode-matched** (K=5): mean F1 = **0.638**, BCa 95 % CI [0.509, 0.727]. The headline result for the methods paper.
- **Cap-lift ΔF1 (cap=100 → cap=1000, bare, all 15)**: +0.177, BCa 95 % CI [+0.055, +0.362]. Lifts concentrated on `lung_cancer` +0.540, `dementia` +0.491, `stroke` +0.281, `hypertension` +0.274 — codelists where the joint retriever pool already contains the gold codes and only the cap was hiding them. cap=500 → cap=1000 is flat (mean ΔF1 ~+0.006 in bare mode), so the bottleneck sits between 100 and 500, not above.

The hierarchy expander (`hierarchy_expander.py`) runs after LLM scoring on queries flagged `include_descendants=True` (T37j) and adds OMOP "Is a" descendants of LLM-included codes, so it can grow the final output beyond the cap. The lift is therefore partial: it recovers descendants of the codes that survived the cap, not the codes the cap itself dropped.

**T37j +0.106 ΔF1 framing at cap=1000.** The T37j K=5 verification at cap=100 (`T37j_path_a_summary.md`) reported mean ΔF1 +0.106 (BCa 95 % CI [+0.049, +0.177]) against the pre-T37i baseline, mode-matched. Re-paired at cap=1000 (pre-T37i baseline taken to equal post-T37j cap=1000 bare-mode by the empirical-equivalence footnote in the summary), mean ΔF1 falls to **+0.070** with BCa 95 % CI **[−0.030, +0.287]** — the CI now straddles zero. The mode-routing benefit shrinks at higher cap because the larger pre-cap pool already surfaces more of the gold codes that the override-mode expander would otherwise have added post-LLM. The T37j lift is real at cap=100 (the production default) but is not statistically distinguishable from zero at cap=1000. The published methods F1 numbers should be read as: cap=1000 gives a ~0.07 F1 absolute lift over cap=100, of which most is structural (the cap itself) rather than the mode-routing innovation.

**Depression precision regression at higher caps.** `depression` regresses in bare mode: F1 0.649 → 0.526 at cap=500 (Δ−0.123), 0.649 → 0.482 at cap=1000 (Δ−0.167). Recall lifts substantially (0.545 → 0.808) but precision drops further because the larger pool surfaces 125 SNOMED depression-adjacent codes the LLM scores `include` that are absent from the narrow NHSD `depr_cod` refset. Diagnostic categorisation (`backend/bench/diagnose_depression_hiv.py`): 86 are clinically-correct depression codes outside the refset scope (e.g. "Single major depressive episode, severe", "Recurrent depression"), 6 perinatal/postnatal, 6 reactive/situational, 6 with psychotic features, 5 personality/chronic, 4 each seasonal / severity-qualifier / anxiety-mixed, 2 bipolar, 2 ICD-10 cross-vocab. **The pattern is a gold/LLM disagreement on refset scope, not an LLM error.** The QOF/NICE `depr_cod` refset intentionally narrows to primary-care quality-of-care monitoring codes; the LLM correctly identifies broader clinical depression. Future work either (a) re-anchors the benchmark gold to a clinically-broader reference, or (b) introduces a per-query reviewer-controlled "include adjacent" toggle so the system can match either narrow or broad scope.

**HIV is retriever-bound, not cap-bound.** `hiv` shows F1 ≈ 0.06 at every cap (0.065 at cap=100, 0.060 at cap=500, 0.048 at cap=1000) because the joint retrievers surface only ~100 candidates for the bare query `"Human immunodeficiency virus (HIV) codes"`, of which only 9 intersect the 243-code NHSD `hiv_cod` refset. The cap never fires; lifting it changes nothing. A targeted retrieval test on the expanded query `"HIV including AIDS-defining illnesses and HIV-related complications"` (`backend/bench/test_hiv_expanded_query.py`) lifted the pre-cap pool from 100 (9 gold) to **427 (all 243 gold)** in a single shot, with 241 of 243 surviving LLM scoring at cap=500 bare — implied F1 ≈ 0.75. **The HIV gold list is recoverable in principle; the current bare-name query under-specifies the clinical scope for refset-style codelists that bundle a primary diagnosis with its AIDS-defining-illness comorbidity codes and its opportunistic-infection codes.** Future work is clinical-taxonomy-aware query expansion: when the parser recognises a refset-style query, the retriever pre-fan-out should issue parallel sub-queries against the well-known associated-condition taxonomy (e.g. NHSD's HIV refset definition, ICD-10's B20–B24 block, the SNOMED HIV-related disorders sub-hierarchy) before merging.

**Production default is unchanged.** `MAX_CANDIDATES` stays at 100 in [`config.py`](./backend/app/config.py). The cap=1000 numbers are the benchmark / scientifically-honest configuration; the live `/api/search` route serves users at cap=100 for latency reasons. The architectural next step is captured in the T38 roadmap ticket (separate config keys for benchmark vs production, move cap after enrichment, LLM-scoring cascade with marginal-gain stopping rule).
